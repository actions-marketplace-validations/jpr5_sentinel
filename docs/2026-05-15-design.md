# Sentinel Design Spec

## Purpose

Deterministic CLI tool that audits GitHub Actions workflows for 20 security dimensions. No AI, no gems — pure Ruby stdlib. Takes a repo or org as input, runs pattern-matching rules, outputs scored findings with line numbers and fix recommendations.

## Input

- `sentinel owner/repo` — fetch via GitHub API
- `sentinel --org owner` — scan all non-archived repos in org
- `sentinel --local /path` — read from disk
- `GITHUB_TOKEN` env var for authenticated API access (optional, anonymous otherwise)
- `--format terminal|json` — output format (default: terminal)
- `--min-severity critical|high|medium|low` — filter findings (default: low, show all)

## Architecture

```
CLI (optparse) → GitHubClient / LocalClient → [raw YAML] → Workflow → RuleEngine → [Finding] → Formatter
```

## Workflow Object

Thin wrapper around parsed YAML. Provides:
- `jobs` — hash of job_id → job_hash
- `steps(job)` — array of step hashes
- `triggers` — the `on:` value (string, array, or hash)
- `permissions(scope: :workflow)` or `permissions(scope: :job, job:)` — permissions hash
- `filename` — basename for reporting
- `raw_lines` — original file as array of strings
- `line_of(pattern)` — finds line number of first match in raw text
- `lines_of(pattern)` — finds all line numbers matching pattern

Line numbers derived from raw text scanning (regex against raw_lines), not YAML parser (which discards position info).

## Finding Data Class

```ruby
Finding = Struct.new(:rule, :severity, :file, :line, :code, :message, :fix, keyword_init: true)
```

Severities: `:critical`, `:high`, `:medium`, `:low`

## Rule Interface

```ruby
module Rules
  class Base
    def name; end        # kebab-case identifier
    def description; end # one-line
    def severity; end    # symbol
    def check(workflow)  # → [Finding]
      []
    end
  end
end
```

## The 20 Rules

### Critical
1. **unpinned-actions** — `uses:` without 40-char hex SHA
2. **shell-injection-expr** — attacker-controllable `${{ }}` in `run:` blocks (github.event.pull_request.title/body, .head.ref, github.event.issue.title/body, github.event.comment.body, github.head_ref, inputs.*)
3. **shell-injection-jq** — `${VAR}` inside double-quoted jq/curl JSON arg strings where VAR originates from attacker-controllable env
4. **dangerous-triggers** — `pull_request_target` + checkout with `ref:` containing `head`

### High
5. **missing-persist-credentials** — `actions/checkout` without `persist-credentials: false` (skip if job does git push)
6. **credential-window** — git config insteadOf appears >5 steps before the push step
7. **static-aws-credentials** — `configure-aws-credentials` with `aws-access-key-id` instead of `role-to-assume`
8. **unscoped-app-token** — `create-github-app-token` without any `permission-` input
9. **docker-build-arg-secrets** — `build-args:` containing `secrets.`
10. **build-publish-same-job** — job has both install/build and publish commands with publish secrets present

### Medium
11. **missing-permissions** — no top-level `permissions:` block
12. **git-config-global** — `git config --global` in `run:` blocks
13. **missing-timeouts** — job without `timeout-minutes`
14. **missing-env-protection** — job with npm publish or id-token: write but no `environment:`
15. **allow-forks-artifact** — `allow_forks: true` on artifact download
16. **missing-frozen-lockfile** — npm/pnpm install without --frozen-lockfile in run

### Low
17. **unpinned-docker-image** — `docker://.*:latest` or `:latest` tag
18. **overly-broad-triggers** — push/pull_request with no branches filter
19. **missing-dependabot** — no dependabot.yml with github-actions ecosystem
20. **missing-zizmor** — no zizmor workflow

## Exit Codes

- 0 — no findings
- 1 — findings at medium/low only
- 2 — findings at critical/high

## Testing Strategy

Scan 3 high-star public repos with many workflows. For each:
1. Run the scanner
2. Manually review findings — classify each as true positive, false positive, or false negative
3. Adjust rules to eliminate false positives and capture false negatives
4. Re-run until stable

Target repos (large, complex, real-world):
- `vercel/next.js` — massive monorepo, many workflows
- `facebook/react` — well-maintained, likely hardened
- `kubernetes/kubernetes` — infra-heavy, complex triggers

## Dependencies

Zero gems. Pure Ruby stdlib:
- `yaml` — YAML parsing
- `net/http` — GitHub API
- `json` — JSON output
- `optparse` — CLI args
- `uri` — URL construction

## Ruby Version

Ruby 3.0+ (for pattern matching, endless methods, Struct keyword_init).
